# 最小博客设计方案

日期：2026-04-09

## 概述

这个项目是一个最小版本的博客系统，使用两个子域名分别承载公开站点和后台管理。

- 公开站点：博客子域名，供访客浏览
- 后台站点：管理子域名，供站长维护内容
- 后端：Spring Boot
- 前端：Vue
- 数据库：MySQL
- 数据库迁移：Flyway

第一版严格控制范围，只包含以下能力：

- 前台首页
- 前台文章详情页
- 后台登录
- 后台文章管理

文章正文以 Markdown 原文存储在数据库中，在前台详情页渲染展示。

## 目标

- 交付一个完整但足够小的博客产品，前台浏览与后台管理职责清晰
- 支持在后台以 Markdown 方式编写和管理文章
- 支持文章以“草稿 / 已发布”的方式流转
- 保持架构足够简单，方便快速开发和部署

## 非目标

- 分类、标签、评论、点赞、搜索、RSS、多作者协作
- 富文本编辑器
- 定时发布
- 媒体库
- 历史版本
- 面向第三方的开放 API

## 推荐架构

推荐采用一个仓库下的三个应用：

- `blog-web`：前台博客 Vue 应用
- `admin-web`：后台管理 Vue 应用
- `blog-api`：Spring Boot 应用，负责认证、文章管理和前台公开查询

这个结构最符合两个子域名的场景，同时把后端逻辑集中在一处，便于维护。

### 选择这个方案的原因

- 与 `blog.<domain>` 和 `admin.<domain>` 的域名结构天然对应
- 前台和后台在代码与职责上彼此隔离
- 两个前端共用一套后端和一套数据库
- 对最小版本来说足够简单，后续也方便扩展

### 备选方案

1. 一个 Vue 前端同时承载前台和后台，再配一个 Spring Boot 后端

这样可以少一个前端项目，但会把两种不同的体验混在一起，不够贴合双子域名结构，后续维护也更容易耦合。

2. Spring Boot 渲染前台页面，Vue 只做后台

这个方案可以做，但会让前端技术路线不统一，对于当前这个最小项目反而增加实现复杂度。

## 域名与部署模型

- `blog.<domain>` 提供前台 Vue 应用
- `admin.<domain>` 提供后台 Vue 应用
- 两个前端都通过 `/api` 调用后端接口
- Nginx 根据子域名分发请求，并把 `/api` 反向代理到 Spring Boot
- Spring Boot 连接 MySQL，并在启动时执行 Flyway 迁移

这个设计不需要单独增加第三个 API 域名，部署结构也更直观。

## 用户侧功能范围

### 前台站点

前台严格只有两个页面：

1. 首页
2. 文章详情页

#### 首页

- 只展示已发布文章
- 展示标题、摘要、发布时间
- 支持点击进入文章详情页

#### 文章详情页

- 通过 slug 加载一篇已发布文章
- 将 Markdown 正文渲染为 HTML
- 如果文章不存在或未发布，返回 404

### 后台站点

后台只包含三个必要视图：

1. 登录页
2. 文章列表页
3. 文章编辑页

#### 登录页

- 单管理员账号密码登录
- 未登录不得访问后台功能

#### 文章列表页

- 展示所有文章，不区分是否发布
- 清晰区分草稿和已发布状态
- 支持新建、编辑、删除

#### 文章编辑页

- 编辑标题
- 编辑 slug
- 编辑摘要
- 编辑 Markdown 正文
- 保存为草稿
- 发布文章

## 认证设计

后台认证采用单管理员模式。

### 推荐机制

- 基于 Spring Security
- 使用服务端维护的会话登录态
- 浏览器通过 `HttpOnly` Cookie 持有认证状态

### 不优先使用 JWT 的原因

当前场景是单管理员后台，不是开放平台。JWT 会额外引入不必要的复杂度，而基于会话的方案更简单、更稳，也更适合登录和退出这样的最小后台需求。

## 数据模型

第一版建议只保留两张核心表。

### `admin_user`

- `id`
- `username`
- `password_hash`
- `created_at`
- `updated_at`

### `post`

- `id`
- `title`
- `slug`
- `summary`
- `content_markdown`
- `status`，取值为 `DRAFT` 和 `PUBLISHED`
- `published_at`
- `created_at`
- `updated_at`

## 数据规则

- `slug` 必须唯一
- `title` 必填
- `content_markdown` 必填
- 新建文章默认状态为 `DRAFT`
- 前台公开接口只能返回 `PUBLISHED` 状态的文章
- 已发布文章如果改回草稿，需要从前台结果中消失
- `published_at` 在文章首次发布时写入

## Markdown 处理方式

- 数据库中保存 Markdown 原文，字段为 `content_markdown`
- 前台详情页负责把 Markdown 渲染成 HTML
- 后台始终围绕 Markdown 原文进行编辑，不引入富文本编辑器

这种方式和“笔记型博客”的需求最一致，也能把第一版复杂度压到最低。

## 接口设计

### 前台公开接口

- `GET /api/posts`
  - 返回已发布文章列表
- `GET /api/posts/{slug}`
  - 根据 slug 返回一篇已发布文章详情

### 后台管理接口

- `POST /api/admin/auth/login`
- `POST /api/admin/auth/logout`
- `GET /api/admin/posts`
- `GET /api/admin/posts/{id}`
- `POST /api/admin/posts`
- `PUT /api/admin/posts/{id}`
- `DELETE /api/admin/posts/{id}`

## Flyway 策略

Flyway 从第一天就必须接入。

### 规则

- 迁移文件放在 Spring Boot 项目内部
- 初始表结构必须通过 Flyway 创建，而不是手动建库建表
- 后续每次结构调整都新增迁移文件
- 已经在共享环境执行过的迁移文件不允许被重写

### 初始迁移范围

第一批迁移至少创建以下内容：

- `admin_user`
- `post`
- 必要索引
- `slug` 唯一约束

### 初始管理员创建方式

初始管理员不通过 Flyway 种子数据写入，而由后端启动初始化逻辑创建。

规则如下：

- Flyway 只负责建表和结构迁移
- Spring Boot 启动时检查是否已有管理员账号
- 如果没有管理员账号，则根据配置的引导凭据创建一个初始管理员
- 引导密码写入数据库前必须先做哈希处理

这样可以保证迁移脚本保持确定性，同时避免把固定管理员凭据写死在 SQL 文件中。

## 发布流程

发布流程保持最小闭环：

1. 管理员新建文章
2. 新文章默认进入 `DRAFT`
3. 管理员可以多次保存草稿
4. 管理员执行发布操作
5. 文章状态变为 `PUBLISHED` 后出现在前台
6. 管理员也可以把已发布文章重新改回草稿，从前台撤下

## 错误处理

### 前台

- 文章不存在时返回 404
- 草稿文章在前台按“不存在”处理，不暴露草稿状态

### 后台

- 未登录访问后台接口时返回 401
- 标题或正文为空时返回明确校验错误
- `slug` 重复时返回明确错误提示

## 测试范围

即使是最小版本，也需要覆盖关键路径。

### 后端

- 登录成功与失败
- 后台文章新增、修改、删除
- 前台接口只返回已发布文章
- 草稿文章不能通过前台访问
- Flyway 迁移可以在启动时正常执行

### 前端

- 前台首页正确渲染文章列表
- 前台详情页正确渲染 Markdown 内容
- 后台登录流程可用
- 后台可以完成文章创建与编辑

## 实现边界

第一版实现时不要突破已经确认的最小范围。

保留：

- 两个前台页面
- 后台登录
- 后台文章管理
- Markdown 存储
- 草稿 / 已发布状态
- Flyway 管理数据库结构

第一版不加入：

- 标签体系
- 搜索
- 分页，除非最基础可用性已经明显需要
- 文件上传
- 多管理员角色

## 本方案已确定的决策

以下决策在实现阶段不再反复讨论，直接按此执行：

- 前端使用 Vue
- 后端使用 Spring Boot
- 数据库使用 MySQL
- 内容以 Markdown 形式存入数据库
- 认证方式为单管理员账号密码
- 文章状态为 `DRAFT` 和 `PUBLISHED`
- 数据库迁移使用 Flyway
- 前台和后台使用不同子域名

## 成功标准

当满足以下条件时，第一版即可视为完成：

- 访客可以打开前台首页并看到已发布文章列表
- 访客可以打开文章详情页并阅读渲染后的 Markdown 内容
- 站长可以登录后台
- 站长可以创建、编辑、发布、转回草稿、删除文章
- 数据库结构由 Flyway 管理
- 系统可以在一个前台子域名和一个后台子域名下顺利部署
